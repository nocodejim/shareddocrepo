DevOps Interview / Sheet 2
Build from Scratch — GitHub · Self-Hosted Runner · JFrog · Rancher
Duration: 15–20 min
Format: Design discussion
Focus: Architecture · Decisions · Trade-offs

Setup for Candidate

Prompt:

We're building a CI/CD pipeline from scratch. The four core pieces you have available are: GitHub for source and workflow orchestration, a self-hosted GitHub Actions runner running inside our private network, JFrog Artifactory as our artifact and container registry, and a self-hosted Rancher cluster for Kubernetes workloads.

Walk me through how you would wire these together — and, just as importantly, why you would make the choices you do. We'll dig into specific pieces as you go.

Let them drive the design first. Use the questions below to probe gaps or test judgment when they stall or skip a layer.

① Trigger
GitHub webhook fires workflow on push/PR/tag. Runner is registered in the repo or org and polls for jobs.
② Build & Publish
Runner builds the container image and Helm chart, then pushes both to JFrog with an immutable tag (Git SHA or semver).
③ Deploy
Runner runs helm upgrade --install against the target namespace. Rancher/K8s pulls the image from JFrog using a pull secret.

Question Bank — Build Decisions

Use these to probe specific layers. Let the candidate design first, then use these questions to fill gaps or pressure-test decisions.

GitHub
How would you structure your GitHub Actions workflow — and why would you make those choices?
Follow-ups: "Would you use a monorepo or separate repos per service?" · "How do you handle environment promotion — same workflow or separate?" · "When does the deploy job run vs. the build job?"
Strong signals
  • Separates build job (produces artifact) from deploy job (consumes it) — with explicit artifact hand-off
  • Uses immutable image tags tied to Git SHA or semver — never latest
  • Environment promotion via separate workflow triggers (tag push, manual approval) rather than re-building
  • Stores secrets in GitHub Actions secrets or an external vault — never in workflow YAML
  • Mentions reusable workflows or composite actions for DRY pipelines across services
Red flags
  • One giant workflow that builds, tests, and deploys to prod in a single job
  • Uses latest tag or dynamic tags that aren't traceable to a commit
  • Hardcodes credentials or tokens in workflow YAML
  • Re-builds the container image for every environment rather than promoting the same artifact
Self-Hosted Runner
Why do we need a self-hosted runner here — and what does that runner need access to?
Follow-ups: "How would you register the runner — repo-level or org-level?" · "What's your runner isolation strategy?" · "How do you handle runner credential sprawl?"
Strong signals
  • Identifies that GitHub-hosted runners can't reach private network resources — that's the core reason for self-hosting
  • Describes what access the runner needs: Docker daemon, JFrog endpoint, kubeconfig/Rancher API, and nothing else
  • Mentions runner isolation: ephemeral runners (new container per job), or at minimum runner groups scoped per environment
  • Stores kubeconfig and registry credentials as GitHub Actions secrets — not baked into the runner image
  • Considers runner lifecycle: how to update, rotate credentials, and scale horizontally
Red flags
  • Can't articulate why a self-hosted runner is necessary — assumes it's just a preference
  • One persistent runner with long-lived credentials baked in — no isolation between jobs
  • Runner has cluster-admin or broad network access because "it's easier"
  • No plan for runner maintenance, patching, or credential rotation
JFrog Artifactory
How would you set up JFrog in this pipeline — what repositories, what access model, and what lives there?
Follow-ups: "How does Kubernetes pull images from JFrog?" · "How do you handle base image hygiene?" · "How would you enforce immutability on published artifacts?"
Strong signals
  • Distinguishes local repos (built artifacts) from remote/proxy repos (upstream deps) and virtual repos (combined views)
  • Configures an imagePullSecret in each namespace referencing a scoped JFrog service account — not a personal credential
  • Enables immutable tags on release repos so published images can't be overwritten
  • Uses JFrog as a proxy/cache for base images — reduces external dependency on Docker Hub or public registries
  • Considers Xray scanning for vulnerability gating before promotion
  • Separate credentials for write (CI runner) vs. read (Kubernetes pull secret)
Red flags
  • Uses a single admin credential for both pushing and pulling images
  • No concept of imagePullSecrets — assumes Kubernetes can reach the registry unauthenticated
  • Allows mutable tags in production — same tag can be overwritten
  • Pulls base images directly from Docker Hub inside the build rather than through JFrog proxy
Rancher / Kubernetes
How would you structure the Rancher cluster — namespaces, RBAC, and how Helm fits in?
Follow-ups: "How does your CI identity get permission to deploy to a namespace?" · "How do you manage Helm releases across environments?" · "What's the difference between a Rancher project role and a Kubernetes RBAC role?"
Strong signals
  • Separate namespaces per environment (app-dev / app-staging / app-prod) — ideally with separate kubeconfigs or service accounts per namespace
  • CI deploy identity has only the RBAC verbs it needs in its target namespace — nothing cluster-wide
  • Helm release names and values files parameterized per environment — one chart, multiple value overrides
  • Uses Rancher projects to group namespaces logically, but verifies actual K8s RBAC with kubectl auth can-i
  • Ingress per namespace; TLS secrets managed in-namespace or via cert-manager
  • Can articulate the difference between Rancher project roles and native K8s ClusterRole/RoleBindings
Red flags
  • Single namespace for all environments — or namespace-per-app without environment separation
  • CI deploy identity has cluster-admin "to keep it simple"
  • Re-builds the Helm chart per environment rather than using value overrides
  • Assumes Rancher UI roles = actual K8s RBAC without verification
  • No plan for secret management — hardcodes app secrets in Helm values
Glue & Judgment
How do these four pieces talk to each other — and where are the failure points you'd design against?
Follow-ups: "How do you trace a production incident back to a specific commit?" · "What's your rollback strategy?" · "If you had to pick the single riskiest part of this pipeline, what would it be?"
Strong signals
  • Can trace: Git SHA → image tag → chart version → Helm release → running pod
  • Rollback means helm rollback to a previous release, not re-building from Git
  • Identifies the runner as the trust boundary: it has credentials to both JFrog and Rancher — and that's what needs the most protection
  • Mentions pipeline gates: lint, test, scan, preflight checks before deploy
  • Describes how to handle a broken deploy without a full pipeline run — e.g., manual helm rollback with restricted access
Red flags
  • Can't trace an artifact version back to a commit
  • Rollback means re-deploying from Git HEAD — no concept of pinned release history
  • No awareness of the runner as a security-sensitive component
  • No gates between build and deploy — any commit goes straight to production

Key Design Decision Quick-Reference

Decision The question to ask Good answer Concern
Image tagging How do you version container images? Immutable tags — Git SHA or semver. Never latest in CI. Uses latest or timestamp tags with no traceability.
Runner access What can the runner reach? JFrog endpoint, kubeconfig for target namespace only, nothing else. Runner has broad network access or cluster-admin by default.
Registry credentials How does Kubernetes pull from JFrog? imagePullSecret per namespace, scoped read-only JFrog token. Shared admin credential or no auth assumed.
RBAC model What can the CI identity do in Kubernetes? Namespace-scoped Role/RoleBinding covering only required verbs and resources. ClusterRole or cluster-admin "to avoid permission issues."
Environment promotion How does an artifact move from dev to prod? Same image tag promoted through environments — no rebuilds. Rebuilt per environment — different artifact in staging vs. prod.
Secrets management Where do app secrets live? External secrets operator, Vault, or sealed secrets — not in Helm values YAML. Plain text in Helm values or committed to the repo.
Rollback How do you recover a bad deploy fast? helm rollback to a pinned prior revision — no pipeline run needed. Rollback requires re-running the pipeline from a previous commit.

Scoring

5
Designs the full pipeline unprompted: immutable tagging, runner isolation, scoped RBAC, pull secrets, environment promotion without rebuilding, and clear rollback path. Identifies the runner as the key trust boundary and proposes gates between stages.
4
Covers all four components with sound decisions. May need a prompt on one area (e.g., secrets management or Rancher vs. K8s RBAC distinction). Trade-off reasoning is clear and defensible.
3
Understands the pipeline flow and most security basics. Has gaps in one or two areas — e.g., vague on imagePullSecret setup or RBAC scoping. Can be coached to the right answer.
2
Knows individual tools but can't connect them into a coherent design. Relies on broad permissions or rebuilds per environment. Security reasoning is reactive rather than proactive.
1
Cannot design beyond one layer. Reaches for cluster-admin, uses latest tags, or conflates runner access with application runtime permissions. No concept of artifact traceability.
