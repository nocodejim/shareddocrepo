DevOps Interview / Sheet 1
Diagnosis & Debugging Screen — Rancher / Helm / GitHub Actions
Duration: 10 min
Format: Scenario-based
Focus: RBAC · Helm · kubectl safety

Scenario

Setup for candidate:

A GitHub Actions deployment dry run for app-dev fails before Helm executes. The workflow authenticates to a private registry, prepares Helm values, and runs a Kubernetes preflight check. The failure is an RBAC error: the preflight requests visibility into kube-system or cluster-level resources, but the deploy identity is intentionally scoped to target-namespace.

Ask: Walk through how you would diagnose this and fix it safely — without casually broadening permissions.

Expected direction: adjust the preflight to namespace-scoped checks and verify the deploy path, not grant cluster-wide access.

Must-Ask Questions P0

P0-1
"Walk me through the path from a GitHub commit to a running pod."
Why it matters
Tests whole delivery chain ownership — not just one tool.
Strong signals
Names trigger → build → chart/image publish → immutable tag → Helm values → namespace → rollout → service → ingress.
Red flags
Jumps to kubectl delete pod; can't link chart values to images; treats Actions as a black box.
P0-2
"The dry run fails before Helm executes — RBAC error mentioning kube-system. What do you check first?"
Why it matters
This is the exact scenario. Tests whether they pinpoint the right failure layer.
Strong signals
Identifies failing command, service account, verb/resource/namespace, and recognizes it's a preflight — not a chart failure.
Red flags
Assumes the image or chart is broken; grants cluster-admin; ignores that Helm never ran.
P0-3
"How should the preflight be changed to fix this safely?"
Why it matters
Tests least privilege in practice, not just theory.
Strong signals
Replaces broad cluster checks with kubectl auth can-i scoped to target-namespace; keeps any cluster-scoped tasks separate.
Red flags
Disables preflight entirely; makes kubectl cluster-info mandatory; requests broad read without a specific resource need.
P0-4
"Which kubectl commands are safe to run during initial diagnosis?"
Why it matters
Tests operational safety — good engineers don't mutate before they understand.
Strong signals
Uses only: get, describe, logs, events, rollout status, auth can-i — always with -n target-namespace.
Red flags
Reaches for apply, delete, rollout restart, or secret dumping first.
P0-5
"How would you debug a private registry image pull failure?"
Why it matters
Tests registry-to-Kubernetes reasoning end to end.
Strong signals
Checks pod events, image name/tag, registry path, pull secret reference, service account annotation, DNS/TLS symptoms, and tag existence.
Red flags
Pulls latest; changes credentials blindly; prints registry password into logs.
P0-6
"What would you explicitly not do in this environment?"
Why it matters
Tests judgment under pressure — separates engineers from cowboys.
Strong signals
Avoids exposing secrets, casual RBAC broadening, mutating prod during diagnosis, deleting pods before reading events, bypassing gates without understanding the failure.
Red flags
Justifies cluster-admin by speed; mentions secret exposure; reaches for destructive commands first.

Nice-to-Have Questions P1

P1-1
"What does helm template prove, and what does it not prove?"
Why it matters
Separates render-time from runtime validation.
Strong signals
Proves chart rendering/values shape — not cluster permissions, image pull success, scheduling, readiness, or app health.
Red flags
Treats template success as deployment success.
P1-2
"How do Rancher roles interact with native Kubernetes RBAC?"
Why it matters
Tests on-prem/Rancher platform awareness vs. vanilla K8s assumptions.
Strong signals
Explains Rancher project/cluster roles may map to K8s permissions but should be verified with kubectl auth can-i as the actual deploy identity.
Red flags
Assumes UI access automatically equals CI service account access.

Signal Summary

✓ Hire Signals
  • Starts with: exact failing command, error text, step ordering, namespace, SA, verb/resource
  • Recognizes failure occurs before Helm — chart changes alone won't fix it
  • Defaults to namespace-scoped checks with -n target-namespace
  • Separates: helm lint → auth can-i → K8s events → image/ingress
  • Proposes rendering manifests to derive required permissions
  • Mentions masking secrets and avoiding -o yaml in shared logs
  • Knows when cluster-scoped permissions are legitimate and when they aren't
✗ Reject Signals
  • "Just give the CI account cluster-admin."
  • Conflates helm dry-run with kubectl preflight
  • Doesn't notice kube-system in the error message
  • Deletes pods or restarts deployments before reading events
  • Prints secret YAML, tokens, or credentials into logs or tickets
  • Can't explain how imagePullSecrets connect to a private registry
  • Accepts latest tags in a controlled deployment pipeline

Preferred Fix — Namespace-Scoped Preflight

  1. Keep the deploy identity scoped to target-namespace — do not broaden it.
  2. Replace broad cluster/kube-system checks with namespace-scoped kubectl auth can-i calls.
  3. Check only the resources the release actually needs in target-namespace.
  4. Render Helm manifests locally inside CI to identify required namespaced resources.
  5. If the chart genuinely requires cluster-scoped resources (CRDs, ClusterRoles), split that into a separately approved admin path.
# Example: namespace-scoped preflight replacing cluster-level checks
kubectl auth can-i get pods -n target-namespace
kubectl auth can-i create deployments -n target-namespace
kubectl auth can-i update services -n target-namespace
kubectl auth can-i get ingress -n target-namespace
kubectl get namespace target-namespace
helm template release-name ./chart -n target-namespace -f values-app-dev.yaml >/dev/null

Scoring

5
Clearly separates CI, registry, Helm, Rancher RBAC, and runtime troubleshooting. Proposes namespace-scoped preflight unprompted. Safe operational judgment throughout.
4
Applies least privilege, fixes the preflight correctly, identifies failure layer. May need a prompt to mention secret safety or Rancher/K8s RBAC distinction.
3
Diagnoses with safe read-only commands and explains Helm/image flow. RBAC reasoning present but surface-level.
2
Understands one tool but misses pipeline and RBAC interactions. May conflate render and runtime validation.
1
Reaches for broad permissions or destructive commands first. Cannot trace a commit to a running pod.
